## 1. Что за проект

**Name:** Bable

**Платформы:** Mobile (Flutter), позже Web.

## 2. Problem Statement

> IT-специалисты не могут эффективно показать свою экспертность.
> Резюме шаблонные и не отражают реальный уровень. GitHub-stars не показывают навык
> коммуникации. Twitter/X не поддерживает технический контент: ломаются threads,
> нет code blocks, лента забита нерелевантным шумом.
>
> Рекрутеры тратят недели на скрининг "профи на бумаге" вместо реальных профи.
>
> **Решение:** социальная сеть для IT, где экспертность доказывается верифицируемым
> контентом — короткими постами с code blocks, оценкой сложности/пользы и
> peer-review от других разработчиков. Профиль = живое портфолио с метриками
> экспертности по темам/языкам.

## 3. Target Users (две равноценные стороны)

### Persona A — IT-специалист
- **Кто:** разработчик / DevOps / QA / data engineer, junior → senior
- **Goal:** Показать реальный уровень и стать видимым для рекрутеров без рассылки CV
- **Pain:**
  - Twitter не для тех-контента (нет syntax highlight, threads рассыпаются)
  - В LinkedIn слишком много шума и "коучей успеха"
  - GitHub показывает код, но не показывает коммуникацию и мышление
  - Сертификаты дорогие, и им мало кто верит

### Persona B — Tech Recruiter / Hiring Manager
- **Кто:** IT-рекрутер, hiring lead, tech founder
- **Goal:** Быстро найти проверенных специалистов с доказанной экспертизой
- **Pain:**
  - Резюме не отражают реальный навык
  - LinkedIn-поиск даёт тех, кто умеет писать резюме, а не код
  - Скрининг — это часы интервью с непрошедшими

## 4. Value Proposition

**Для разработчика:**
*"Покажи экспертность контентом, а не должностями в резюме.
Каждый твой пост, оценённый сообществом, повышает твой Expertise Score
по конкретным темам. Рекрутеры найдут тебя сами."*

**Для рекрутера:**
*"Ищи специалистов по верифицированным навыкам, а не по словам в резюме.
Видишь не лайки, а оценённые экспертами публикации с уровнем сложности."*

## 5. Core Concepts (что отличает от Twitter)

| Концепт | Описание |
|---------|----------|
| **Post** | Короткий пост (≤1000 символов) с типом (Insight / Solution / Question / Project), уровнем сложности, тегами языков/тем, поддержкой code blocks |
| **Endorsement** (вместо Like) | Голос с весом, зависящим от Expertise Score голосующего по этой теме. Senior-голос > junior-голос |
| **Expertise Score** | Динамические показатели по темам и языкам (например: `Rust 820`, `React 540`, `System Design 700`). Растёт от endorsements, падает от устаревания |
| **Profile = живое CV** | Не bio, а портфолио: посты, проекты, expertise по навыкам, badges, история активности |
| **Recruiter view** | Отдельная роль с поиском по навыкам, уровню, доступности; saved candidates; outreach |
| **Trust model** | Peer-review (Stack Overflow style): экспертность доказывается голосами других экспертов |

## 5.1 Expertise Score (механика)

Score хранится **по теме / языку / навыку**, не общий. Пример профиля:

```
Alice
├── Rust            820 ▮▮▮▮▮▮▮▮▯▯
├── React           540 ▮▮▮▮▮▮▯▯▯▯
├── System Design   710 ▮▮▮▮▮▮▮▯▯▯
└── PostgreSQL      390 ▮▮▮▮▯▯▯▯▯▯
```

### Формула (MVP, упрощённо)

```
score(user, topic) = Σ endorsement.weight × complexity_multiplier  −  decay
```

| Параметр | Значение | Зачем |
|----------|----------|-------|
| `endorsement.weight` | `log10(1 + endorser.score_on_same_topic)` | Senior-голос весит больше junior'а в той же теме (StackOverflow-style) |
| `complexity_multiplier` | `easy=1.0`, `medium=1.5`, `hard=2.5` | Сложные посты дают больше очков (LeetCode-style) |
| `decay` | −2% / месяц без новых постов в теме | Эксперт без активности теряет актуальность |
| floor | `0` | Нет отрицательных очков |
| starting score | `0` | Новички начинают с нуля (решение про onboarding — позже) |

### Принципы дизайна Score

1. **Прозрачность.** Пользователь должен видеть, **почему** счёт такой
   (за какие посты, кто эндорсил, сколько весил каждый endorsement)
2. **Невзламываемость.** Нельзя накрутить ботами: вес зависит от уже-полученных endorsements
3. **Контекстность.** Score всегда показывается с темой — `Rust 820`, не просто `820`
4. **Сравнимость для рекрутера.** Можно фильтровать "Rust ≥ 700"

### Что Score НЕ показывает

- Общий "рейтинг человека" — отказываемся от единого числа, оно врёт
- Количество постов — это про объём, не качество
- Подписчики — это про популярность, не экспертность

## 6. Content Format (MVP)

- **Короткие посты** — основа, но с code blocks и тегами
- **Лимит:** 256 символов (без code block; code block считается отдельно — TBD)
- Длинные статьи и Q&A — out of scope MVP, на будущее

## 7. MVP Scope

### In scope (Phase MVP)
- Регистрация / вход (developer + recruiter роли)
- Создание поста с типом, тегами, code block
- Endorsement (взвешенный)
- Feed (хронологический, без алгоритма)
- Профиль с expertise score и постами
- Поиск: для рекрутера — по навыкам/уровню; для всех — по тегам
- Базовая навигация

### Out of scope (потом)
- Длинные статьи (Medium-формат)
- Q&A с принятыми ответами (Stack Overflow-формат)
- Видео / стримы
- Бейджи и геймификация уровней
- LeetCode-стиль challenges
- Сообщения / чаты
- Уведомления (push)
- Algorithmic feed
- OAuth GitHub / внешние proof-of-work

## 8. Success Criteria (для учебного проекта)

- Пройден полный флоу: brief → research → tokens → components → screens
- Дизайн-система выведена из проблемы, а не скопирована
- AI может сгенерировать новый экран по правилам без дополнительных уточнений
- Дизайнер может за 1 минуту понять, что AI сгенерировал не так

## 9. Constraints & Principles

| Принцип | Почему |
|---------|--------|
| Контент важнее хрома | Tech-контент — суть продукта. UI не должен с ним соревноваться |
| Доверие к голосам, не к числам | Не показываем лайки сами по себе, показываем взвешенную репутацию |
| Минимум дофамин-крючков | Аудитория IT не терпит игровой UI. Серьёзный, не "социальный" тон |
| Editorial light по умолчанию, dark доступен | См. `02-strategy/CHOSEN-DIRECTION.md` — editorial calm требует light. Dark — доступен в settings |
| Mobile-first | Сначала Flutter |
